# ServerStick Security Audit — v0.1 Alpha

**Date:** 2026-05-20  
**Scope:** Preseed template, bootstrap script, Pi Agent, secrets model, tunneling, Docker services, physical distribution.  
**Assumption:** Threat model is "privacy-minded consumer homelab", not nation-state. We still design against opportunistic attackers, compromised infrastructure, and supply chain risks.

---

## Executive Summary

| Component | Risk | Notes |
|---|---|---|
| Preseed ISO distribution | 🔴 **HIGH** | Starter keys visible in initrd; hardcoded default password; no integrity verification on bootstrap |
| First-boot network exposure | 🟡 **MEDIUM** | `curl \| bash` with no signature; downloads binaries without checksums |
| Secrets at rest | 🟢 **LOW** (if SOPS implemented) | age key generation on-device is correct; SOPS-encrypted files are solid |
| Secrets in transit | 🟡 **MEDIUM** | TLS everywhere, but bootstrap fetches over HTTPS with no cert pinning |
| Docker containers | 🟡 **MEDIUM** | Host networking; no user namespace remapping; root-owned Docker socket |
| Pi Agent privilege | 🟡 **MEDIUM** | Agent is the sysadmin; root is correct for a dedicated appliance. Real security is input validation + audit logging, not privilege theater |
| Tunnel / remote access | 🟢 **LOW** | Pangolin WireGuard outbound-only; no inbound ports punched |
| Matrix federation | 🟡 **MEDIUM** | Federation = attack surface; bridges = reverse-engineered protocols with unknown bugs |
| LLM / prompt injection | 🟢 **LOW** (with architecture fix) | Must never fetch skill docs remotely; agent must not have access to decrypt secrets in chat context |
| Physical security | 🟡 **MEDIUM** | Stick contains installer only (no secrets), but recovery card in box could expose Hetzner username |
| Updates | 🔴 **HIGH** | No signed update mechanism defined; `curl \| bash` update path is identical to initial install |

## 1. Preseeded ISO (Distribution Layer)

### 1.1 Starter Key Visibility in initrd

**Finding:** The preseed template embeds `%%STARTER_KEY%%` which is substituted during ISO build. The preseed file goes into the initrd (initramfs), which is an uncompressed cpio archive inside the ISO. Anyone can:

```bash
# Extract and grep in ~3 seconds
7z x serverstick.iso -oiso/
cpio -id < iso/initrd.img
grep -r "sk-" iso/  # finds starter key
```

**Impact:** Starter keys are ~20 credits, low value. But if the build script ever includes a real provisioned key (accidentally), catastrophic. Also, attacker can farm starter keys by buying sticks, extracting keys, and reselling sticks.

**Mitigation:**
- Starter key should be generated by the **cloud API at first boot**, not baked into the ISO.
- Alternative: Each ISO gets a unique **one-time provisioning token** (not an API key). The device calls the cloud API with the token and its hardware fingerprint; cloud returns a starter key bound to that device. Token is burned after use.
- Preseed contains only a **non-secret provisioning URL** or a **public key fingerprint**.

### 1.2 Hardcoded Default Password

**Finding:**
```
d-i passwd/user-password password serverstick
d-i passwd/user-password-again password serverstick
```

The preseed file contains the literal string `serverstick`. Anyone who extracts the preseed learns the default password. The comment says "First boot forces password change" — but that requires the user to log in, which they may not do for a headless appliance.

**Impact:** If the user never logs in interactively, the default password persists. SSH is enabled (`tasksel/tasks-multiselect select ssh-server`). Anyone on the LAN can `ssh serverstick@serverstick.local` with password `serverstick`.

**Mitigation:**
- Generate a **random password per ISO** (like router firmware does). Print it on the sticker in the box.
- Or: Disable password auth entirely; use SSH key only. But that's unfriendly for recovery.
- Or: Force password change on **first SSH login** (not just "first boot" which is headless).
- **Best:** No interactive login needed. All management is through the Pi Agent web UI. Disable SSH password auth by default; enable only with physical confirmation (button press / jumper / USB auth key).

### 1.3 No Integrity Verification on Bootstrap

**Finding:**
```
curl -sL https://get.serverstick.sh | SERVERSTICK_STARTER_KEY=... bash
```

No GPG signature, no checksum, no TLS pinning. Compromised CDN, DNS hijack, or compromised GitHub account = immediate root on every new device.

**Impact:** Complete takeover of every new ServerStick installation. Supply chain attack at scale.

**Mitigation:**
- Bootstrap script must be **signed** (ed25519 detached signature). The preseed embeds the public key (or a hash of it). The late_command verifies before executing.
- Even better: Bootstrap script is **embedded in the ISO** (not fetched). The ISO itself is the distribution. If the ISO is compromised, game over anyway — fetching the bootstrap over the network just adds another attack surface.
- **Best:** Embed the entire bootstrap in the ISO. No network fetch for the installer script. Only Docker images and Newt binary are fetched, and those have their own checksums / Docker Content Trust.

### 1.4 Log Shredding Inadequate

**Finding:**
```
in-target shred -u /var/log/syslog 2>/dev/null || true
```

`shred` on journaling filesystems (ext4 with journal) does not reliably erase data. The journal may contain the preseed arguments (which include the starter key). Also, `shred -u` doesn't clear the inode metadata reliably.

**Impact:** Forensic recovery of starter key from disk.

**Mitigation:**
- Mount `/var/log` as a **tmpfs** during install, so logs never touch disk. Or:
- Use `wipefs` + zero the partition after install. Or:
- Don't put secrets in preseed logs in the first place — pass them via a one-time file that's `shred`+`sync`+`rm` before logging starts.

### 1.5 No Disk Encryption

**Finding:** Preseed does not configure LUKS. All data at rest is plaintext.

**Impact:** Stolen laptop = all data readable. SOPS-encrypted secrets are safe (need age key), but Docker volumes, SQLite databases, and config files are plaintext.

**Mitigation:**
- **v1:** Accept the risk. Homelab threat model; user can enable LUKS manually. SOPS protects the real secrets.
- **v2:** Offer optional full-disk encryption with password-derived LUKS key (same password as SOPS/restic).
- **Partial mitigation:** Encrypt `/var/lib/serverstick/data` as a LUKS volume or EncFS. But adds complexity.

## 2. Bootstrap Script (First-Boot Execution)

### 2.1 Recursive `curl | bash`

**Finding:** The bootstrap script itself uses `curl | bash` / `curl | sh` three times:
1. `https://get.docker.com | sh`
2. `https://deb.nodesource.com/setup_22.x | bash`
3. `https://github.com/pangolin-oracle/newt/releases/latest/download/newt-linux-amd64` (binary, no checksum)

**Impact:** Same as 1.3 — any compromised endpoint = root. The Docker install script alone is a massive attack surface.

**Mitigation:**
- **Docker:** Use Debian's `docker.io` package (`apt install docker.io`). Slightly older, but signed by Debian. Or pin the Docker CE repo GPG key and version.
- **Node.js:** Use Debian's `nodejs` and `npm` packages, or download a specific Node.js binary with SHA256 verification.
- **Newt:** Download a **specific release** (not `latest`) and verify SHA256 against a hardcoded hash in the bootstrap.

### 2.2 Git Clone Without Verification

**Finding:**
```bash
git clone --branch "${BRANCH}" --depth 1 "${REPO}" "${SS_OPT}"
```

No GPG signature verification on tags/commits. No pinning of the repo URL. `BRANCH` and `REPO` come from environment variables.

**Impact:** Environment variable injection = clone from attacker's repo.

**Mitigation:**
- Pin a **specific commit hash** in the bootstrap script, not a branch name.
- Or embed the source code directly in the ISO.
- Verify GPG signatures on tags if using Git.

### 2.3 Agent.env Contains Starter Key in Plaintext

**Finding:**
```bash
cat > "${SS_DIR}/agent.env" << EOF
SERVERSTICK_STARTER_KEY=${STARTER_KEY}
EOF
```

The systemd env file contains the starter key in plaintext. It's chmod 600, but root can read it (and so can any process running as root, which is... everything in this bootstrap).

**Impact:** Any container breakout or process running as root can read the starter key. Low value (~20 credits), but leaks usage info.

**Mitigation:**
- Don't store the starter key in an env file. Pass it directly to the Pi Agent on first startup, then delete it. Or:
- SOPS-encrypt it. But the SOPS key isn't generated until first boot. Chicken-and-egg.
- **Best:** The starter key is ephemeral. Use it once during `/api/setup` to call the cloud, then overwrite with the real API key (SOPS-encrypted). The plaintext starter key only exists for ~60 seconds during first boot.

### 2.4 Pangolin.env Bug (Credential Loss)

**Finding:**
```bash
cat > "${SS_DIR}/pangolin.env" << ENVPANG
NEWT_SECRET=***
ENVPANG
```

This writes the literal string `***` instead of the actual secret. But also, even if fixed, the secret is in plaintext in an env file.

**Impact:** Newt can't connect (literal `***` is wrong). If fixed, secret is on disk in plaintext.

**Mitigation:**
- Fix the bug (use `"${NEWT_SECRET}"` not `***`).
- SOPS-encrypt `pangolin.env` after setup.
- The systemd service should use `sops exec-env` to decrypt at startup.

## 3. Pi Agent (Runtime)

### 3.1 Privilege Model

**Finding:** The Pi Agent runs as a systemd service. The bootstrap doesn't create a dedicated user. It runs as whatever user systemd defaults to (likely root, since it needs Docker access).

**Reality Check:** The Pi Agent IS the sysadmin of this appliance. It needs to `systemctl`, `docker compose`, edit configs, install packages, and yes — `rm` things. A sudo whitelist that blocks `rm` is hiring a sysadmin and telling them they can't delete files. `docker` group access is already root-equivalent (`docker run -v /:/host -it alpine chroot /host sh`). Pretending a `serverstick` user is "safer" is security theater.

**Honest Threat Model:**
- Physical theft → disk encryption (optional for v1; it's a homelab appliance)
- Remote compromise → agent only listens on localhost + authenticated Pangolin tunnel
- LLM going rogue → this is the real threat, but privilege separation doesn't help; a compromised LLM with `docker` access can already own the box

**Actual Security for v1:**
- **Agent runs as root** (or unrestricted sudo — same thing). It's a dedicated appliance.
- **Input validation, not privilege separation.** The LLM never generates raw bash. All actions go through skill templates with defined, auditable operations.
- **Audit logging.** Every agent action logged to `/var/log/serverstick/agent-actions.log`.
- **Authenticated access only.** Web dashboard behind session auth; no anonymous LAN access.
- **Secrets out of context.** Agent reports "configured/missing", never shows decrypted values in chat.

### 3.2 SOPS Key Generation Timing

**Finding:** The age key is supposed to be generated "on-device at first boot" (per PLAN.md), but the bootstrap script doesn't generate it. It's left for `/api/setup` or manual setup.

**Impact:** Secrets may sit in plaintext until the user completes setup. If setup is delayed, attack window.

**Mitigation:**
- Generate the age keypair **in the bootstrap script**, immediately. `age-keygen -o /etc/serverstick/sops/age.key`.
- chmod 600. Don't wait for the user to click through a wizard.

### 3.3 LLM Router and Secrets in Context

**Finding:** The Pi Agent uses an LLM (DeepSeek / GLM) for chat, troubleshooting, and service management. If the LLM context includes system prompts that have access to secret values, prompt injection or jailbreak could leak secrets.

**Impact:** User asks "why isn't my tunnel working?" and the LLM helpfully outputs the full `pangolin.env` contents, including `NEWT_SECRET`.

**Mitigation:**
- **NEVER pass decrypted secrets into the LLM context.** The agent can read SOPS files to check if a key exists, but it must not decrypt and show values.
- Service status should report "configured" vs "missing", not the actual value.
- If troubleshooting requires showing a config, redact secrets: `NEWT_SECRET=sk-...***`.
- The `sops --extract` operation should only be used by the **systemd service startup scripts**, not by the LLM reasoning path.

### 3.4 Never Fetch Remote Skill Docs

**Finding:** The current architecture (YAML catalog + generic `SkillBase`) is correct. But we must **formally prohibit** ever fetching markdown/docs into the LLM context from remote.

**Mitigation:**
- Add this as a **security invariant** in the codebase: "No LLM context may contain text fetched from remote URLs."
- Catalog updates are YAML data only, signed with the baked-in ed25519 key.
- Help text / documentation goes to the Svelte dashboard (client-side), never into the agent's brain.

## 4. Docker Services

### 4.1 Host Networking

**Finding:** `docker-compose.yml` uses host networking for all services.

**Impact:** Containers bind directly to host interfaces. No network isolation between services. A compromised container can reach any other service's port on localhost.

**Mitigation:**
- **v1:** Accept the risk. Host networking is required for mDNS and some services.
- **v2:** Use a dedicated Docker bridge network for the service stack. Only expose necessary ports to the host. Keep Matrix on its own network.

### 4.2 Docker Socket Mount

**Finding:** Watchtower (and potentially other services) mount `/var/run/docker.sock`.

**Impact:** Any container with the Docker socket has root-equivalent access. Watchtower is a known risk vector.

**Mitigation:**
- Run Watchtower with read-only socket access if possible.
- Or: Replace Watchtower with a Pi Agent-managed update check. The agent pulls updates via skill template, not a container with socket access.
- Label-based filtering (`WATCHTOWER_LABEL_ENABLE=true`) is already in place — good.

### 4.3 Container Image Provenance

**Finding:** Docker images come from `ghcr.io`, `frooodle`, `privatebin`, `lscr.io`, `louislam`, `amir20`, `danielgatis`, `containrrr`, `matrix-org`, `vectorim`. No image digest pinning.

**Impact:** Compromised upstream registry = malicious container. Tag `latest` is mutable.

**Mitigation:**
- Pin to **specific digests** in the compose file:
  ```yaml
  image: frooodle/s-pdf@sha256:abc123...
  ```
- Or maintain a private registry mirror with verified images.
- At minimum, pin to specific version tags, not `latest`.

## 5. Tunnel / Remote Access

### 5.1 Pangolin Architecture

**Finding:** Newt connects **outbound** via WireGuard to Pangolin's Gerbil relay. No inbound ports opened.

**Assessment:** This is the correct security model. Even if the device is fully compromised, the attacker can't open new inbound ports — they can only use the existing tunnel.

**Risk:** Pangolin Cloud itself could be compromised. If the attacker controls Pangolin, they can route traffic to the device. But this is true of any tunnel provider.

**Mitigation:**
- Document that users can switch to self-hosted Pangolin if they don't trust the cloud.
- Monitor for unexpected Pangolin resource changes from the device side.

### 5.2 Newt Binary Provenance

**Finding:** Bootstrap downloads Newt from GitHub releases (`latest`) with no checksum verification.

**Impact:** Compromised Newt binary = attacker controls the tunnel. Can intercept all traffic.

**Mitigation:**
- Pin to a specific Newt version and SHA256.
- Or: Include Newt binary in the ISO itself.

## 6. Matrix / Federation

### 6.1 Federation Attack Surface

**Finding:** Federation is enabled by default. The device is accessible at `matrix.{device}.serverstick.com`.

**Impact:** Any Matrix server on the internet can connect to this device. Synapse has had CVEs. Federation exposes the device to unsolicited traffic.

**Mitigation:**
- **v1:** Accept the risk. Federation is the value prop. Keep Synapse updated (Watchtower or agent-managed).
- Monitor for abnormal federation traffic patterns.
- Consider fail2ban or rate-limiting on the Pangolin side.

### 6.2 Bridge Security

**Finding:** Bridges (WhatsApp, Discord, Telegram, Signal) are reverse-engineered against undocumented protocols. They run as Docker containers with host networking.

**Impact:** Bridge bugs = leaked messages, unauthorized access to chat accounts, or device compromise.

**Mitigation:**
- Sandbox bridges in their own Docker network (already planned: `serverstick-matrix` bridge).
- Run bridges as unprivileged users inside containers.
- Pi Agent monitor: auto-restart on crash, alert on CPU/memory spikes.

## 7. Physical / Distribution

### 7.1 USB Stick Contents

**Finding:** The USB stick contains a Debian installer ISO. No secrets. After install, the stick is reformatted or can be removed.

**Assessment:** Correct. The stick is an installer, not a secret store. Good.

### 7.2 Recovery Card

**Finding:** Sticker in box has Hetzner Storage Box username (non-secret identifier) and a blank line for password hint.

**Impact:** If the user writes their actual password on the card, physical theft = total compromise. But this is user error, not a design flaw.

**Mitigation:**
- Print **"DO NOT WRITE YOUR PASSWORD HERE"** on the card.
- The password hint should be something only the user understands ("first dog's name + birth year"), not the password itself.

### 7.3 Supply Chain (Manufacturing)

**Finding:** Who builds the ISOs? If the build machine is compromised, every stick is compromised.

**Mitigation:**
- Build ISOs in CI/CD (GitHub Actions) with reproducible builds.
- Publish ISO checksums (SHA256) on the website.
- Sign ISOs with GPG (detached signature).

---

## 8. Updates

### 8.1 No Defined Update Mechanism

**Finding:** The bootstrap script can update via `git fetch && git reset --hard`, but there's no signature verification, no rollback mechanism, and no staged rollout.

**Impact:** Compromised update = instant root on all devices.

**Mitigation:**
- **v1:** Manual updates only. User clicks "update" in dashboard; agent downloads signed package and verifies.
- **v2:** A/B updates with rollback. Or use `ostree` / `rpm-ostree` for atomic updates.
- At minimum: agent code updates must be signed with the project's ed25519 key.

## What's Safe to Bake into the Preseed/ISO

✅ **Safe:**
- Debian netinst base
- Preseed partitioning, locale, basic packages
- Docker (from Debian repos, not `get.docker.com`)
- Systemd service definitions
- The bootstrap script itself (embedded, not fetched)
- A **public** ed25519 verification key (for signature checks)
- Generic `serverstick` user (with random password)
- `logind.conf` settings (lid close ignore)
- avahi-daemon for mDNS

❌ **Never bake:**
- API keys (starter or earnings)
- Newt credentials
- age private keys
- SOPS-encrypted files
- Any secret or credential

🟡 **Bake with caution:**
- Newt binary (include it, but verify its provenance)
- Node.js binary (or use Debian packages)
- Pi Agent source code (or embed the git repo)
- Service catalog YAMLs (these are public data, but must be signed)

## Threat Actor Scenarios

### Scenario A: Attacker buys a stick, extracts starter key, mass-resells
- **Risk:** Medium (starter key is only ~20 credits)
- **Current mitigation:** None
- **Fix:** One-time provisioning tokens instead of baked keys.

### Scenario B: Attacker compromises `get.serverstick.com`
- **Risk:** Critical (every new install is root'd)
- **Current mitigation:** HTTPS only
- **Fix:** Embed bootstrap in ISO; no network fetch for the installer.

### Scenario C: Attacker compromises GitHub repo, pushes malicious agent code
- **Risk:** Critical (all devices update to malware)
- **Current mitigation:** None
- **Fix:** Signed updates with pinned commits; code review; no auto-update without user confirmation.

### Scenario D: Attacker steals the physical laptop
- **Risk:** Medium (data at rest is mostly plaintext; SOPS secrets are safe without age key)
- **Current mitigation:** SOPS encrypts secrets
- **Fix:** Optional full-disk encryption; age key is on the disk but SOPS-encrypted files are useless without it.

### Scenario E: Attacker jailbreaks the Pi Agent LLM
- **Risk:** Medium (agent has Docker + systemd access)
- **Current mitigation:** None formalized
- **Fix:** Security invariant: secrets never enter LLM context. Agent operates on "configured/missing" boolean state, not decrypted values.
